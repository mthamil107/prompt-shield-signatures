# prompt-shield-signatures — Schema v1

This repo publishes a federated threat-intel feed for the [prompt-shield](https://github.com/mthamil107/prompt-shield) prompt-injection firewall.

- **Source of truth:** one YAML file per signature under `v1/source/sig-NNNN.yml`.
- **Published artifact:** a single merged JSON file at `v1/signatures.json`, regenerated by CI on every merge to `main`.
- **Client behavior:** firewalls fetch `v1/signatures.json` daily and load every entry as a detection rule. (v0.7.0 moves to hourly via Sigstore Cosign; for v0.6.0-alpha the signing cadence sets the upper bound — see [`../THREAT-MODEL.md`](../THREAT-MODEL.md).)

If you are contributing a new signature, you only ever need to write one YAML file. CI handles the rest.

## File layout

```
v1/
  source/
    sig-0001.yml
    sig-0002.yml
    ...
  signatures.json          # generated; do not edit by hand
schema/
  schema.json              # JSON Schema for a single signature
  signatures-schema-v1.json# JSON Schema for the published envelope
  SCHEMA.md                # this file
  example-valid.yml
  example-invalid.yml
```

## Required fields

Every signature YAML file MUST include all of the following.

### `id` — string, `^sig-[0-9]{4}$`
Stable identifier, e.g. `sig-0042`. IDs are **immutable** once published — never reuse or renumber. Pick the next free integer in `v1/source/`.

### `type` — enum
How the pattern is matched at runtime.

| value | meaning |
|---|---|
| `regex` | Python `re`-compatible regular expression. Must compile. |
| `substring` | Case-insensitive literal substring search. |
| `semantic_hash` | 16-hex-character 64-bit SimHash fingerprint of the canonical attack text. |

### `pattern` — string, 1–4096 chars
The matcher payload itself. Format depends on `type` (see above). For `regex` patterns, prefer `(?i)` flags and `\b` boundaries over greedy `.*` to keep false-positive rates manageable.

### `category` — closed enum
Coarse-grained taxonomy. **Must be one of:**

`instruction_override`, `dan_jailbreak`, `indirect_injection`, `system_prompt_leak`, `role_play_bypass`, `encoding_smuggling`, `tool_misuse`, `data_exfiltration`, `policy_evasion`, `context_overflow`, `multilingual_bypass`, `obfuscation`.

If your attack genuinely does not fit any of these, open a schema-change PR — do not invent a value. Adding a category is a schema bump; mislabeling is a silent dashboard bug.

### `severity` — enum
Suggested action tier for clients:

| value | client default |
|---|---|
| `critical` | block + alert |
| `high` | block |
| `medium` | flag / score |
| `low` | log only |

### `attack_class` — closed enum
**How** the attack is delivered, orthogonal to `category` (which is **what** the attacker wants).

`direct_injection`, `multi_turn`, `indirect_injection`, `multilingual`, `encoded`, `adversarial_suffix`, `social_engineering`.

### `source` — object
Provenance for auditability.

```yaml
source:
  origin: garak             # one of: garak | owasp | community | adversarial-fatigue | anthropic-paper
  reference: "garak.probes.promptinject.HijackHateHumans"   # URL, probe id, paper §, PR/issue link
```

Both fields are required. No extra fields allowed under `source`.

### `first_seen` — date `YYYY-MM-DD`
ISO-8601 date the attack was first observed or published. Immutable.

### `description` — string, 8–200 chars
One-line human summary. Hard-capped at 200 chars so it renders cleanly in dashboards, alert payloads, and audit logs.

---

## Optional fields

| field | type | default | notes |
|---|---|---|---|
| `confidence` | number `0.0`–`1.0` | `0.9` | Curator's belief this pattern is a true positive in the wild. Clients may fold this into their own scoring. |
| `tags` | array of strings, ≤16, unique | — | Lowercase, snake/kebab-case, 2–32 chars (`^[a-z0-9][a-z0-9_-]{1,31}$`). For facets the closed enums do not cover. |
| `last_updated` | date `YYYY-MM-DD` | — | Optional on first publish; CI may auto-stamp on edit. |
| `notes` | string, ≤2000 chars | — | Long-form curator notes: caveats, FPR risks, rollout guidance. |
| `language` | ISO 639-1 code (`^[a-z]{2}$`) | — | Set **only** for non-English variants (e.g. `fr`, `ja`). Absence implies English / language-agnostic. |

## Top-level published file (`v1/signatures.json`)

CI emits exactly this shape:

```json
{
  "$schema": "https://raw.githubusercontent.com/mthamil107/prompt-shield-signatures/main/schema/signatures-schema-v1.json",
  "version": "1",
  "generated_at": "2026-06-24T12:00:00Z",
  "signature_count": 137,
  "signatures": [ { /* one entry per schema.json */ }, ... ]
}
```

- `generated_at` MUST be RFC3339 UTC with a trailing `Z`.
- `signature_count` MUST equal `len(signatures)`. Clients SHOULD assert this on fetch.
- `version` is currently `"1"`. A breaking schema change ships under `v2/`.

## Design rationale

**Why YAML at source, JSON at the edge?**
Contributors write YAML because it supports comments, block scalars (`|`) for multi-line notes, and is forgiving about quoting. Clients consume JSON because every language ships a fast, schema-validated JSON parser by default. CI bridges the two so neither side is inconvenienced.

**Why is the ID format fixed to `^sig-[0-9]{4}$`?**
- Sortable as strings.
- Greppable in logs and PR titles.
- 4 digits = 10 000 slots, enough headroom for the foreseeable future without bloating identifiers.
- The fixed prefix `sig-` namespaces our IDs so they cannot collide with CVE, CWE, or vendor IDs in downstream telemetry.

**Why are `category` and `attack_class` closed enums (not free-text)?**
This was the main tradeoff. Free-text would lower contributor friction but would silently fragment dashboards (`jailbreak` vs `jailbreaking` vs `dan-jailbreak`) and break per-category metrics within weeks. We chose closed enums and accepted that adding a new value requires a schema PR. Use `tags` for anything finer-grained — that field is intentionally free-form (within a charset constraint).

**Why is `description` capped at 200 chars?**
It must fit in a single line of an alert payload, a Slack notification, and a CLI log row without truncation surprises. Long-form context goes in `notes`.

**Why `additionalProperties: false` everywhere?**
Typos in field names (e.g. `severtiy:`) would otherwise silently be ignored and the entry would publish without the intended severity. Strict rejection forces the typo to surface in CI.

**Why are dates `YYYY-MM-DD` and not `date-time`?**
`first_seen` is a curatorial claim, not a wall-clock event — sub-day precision is false precision. The envelope's `generated_at` is a real event and uses full RFC3339.

## Validating locally

```bash
pip install jsonschema pyyaml
python - <<'PY'
import json, yaml, jsonschema
schema = json.load(open("schema/schema.json"))
doc = yaml.safe_load(open("v1/source/sig-0001.yml"))
jsonschema.validate(doc, schema)
print("OK")
PY
```

CI runs the same check on every PR plus a cross-file uniqueness check on `id`.
